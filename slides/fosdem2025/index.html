Current state of geospatial data

Outdoor data

Indoor data

The current state of outdoor geospatial data is service-centric. This means that you can easily access the data you need through predefined services. For outdoor data, this is straightfoward as the data is well known and in most cases publicly available. Some services may provide more detailed information, but regardless on which service you use, you are always able to access some form of data pertaining to a location, even if it is not as detailed as you may want it to be.

Current state of geospatial data

Outdoor data

Indoor data

However, when it comes to indoor data, the situation is quite different. While there are services that collect or crowdsource geospatial data of indoor environments, they are not yet as complete as their outdoor counterparts. Furthermore, the collection of indoor environments is very scattered among different services. In a way, this makes sense, since indoor environments are more complex and diverse than outdoor environments. Users or building owners may not want to share their data with the public, which makes it harder to collect and share indoor geospatial data and to govern who publishes this data. But there are also some other reasons to consider, and one of those reasons is that there are a lot of standardisations to represent buildings or indoor environments in general. This makes it hard to create a service that can provide you with indoor data in the same way that you might expect.

Current state of positioning systems

Outdoor

Indoor

A similar story can be told about positioning systems. Outdoor positioning systems are well known and widely used. The Global Positioning System (GPS) is a prime example of this. If we are outdoors, we can easily determine our location using this single technology. That is not to say that there are no other technologies that can be used to determine your location outdoors, but GPS is by far the most widely used and standardised technology. Again, indoors the situation is quite different. There are several technologies that can be used to determine your location indoors, but none of them are as widely used or standardised as GPS. This makes it hard for developers to create applications that work indoors as well as outdoors and often leads to navigation applications that only work for one particular building, framework or event. Additionally, these positioning techniques often require knowledge about the building layout and infrastructure, which again leads to the original problem that this information is scattered among services.

Global data & service discovery

Find data globally about a particular location

In general, we have to types of data or service discovery. We can have global discovery methods such as geocoding services that allow you to find information about a particular location without being at that location. It is by far one of the most common ways to discover data about a location and is used in many applications that require information about a given location. This type of discovery works when you are near a location, but it is not one of the requirements for this type of discovery.

Local data & service discovery

Find data locally at a particular location

The second type of data discovery is also the most important type of data discovery for indoor environments. Local data discovery is about finding data at a location. With this type of discovery we assume we are near the location that provides the data pertaining to that location.

Geospatial-centric service discovery

The future of geospatial services lies in its discovery through the very locations it represents, rather than predefined services.

My vision is to create a world where geospatial services are discovered through the very locations they represent, rather than using predefined services. Or in other words, decentralised geospatial data discovery. This means that when you are in a particular location, you can discover the services that are available in that location. Simply said, you could assume that you do a kind of geospatial query on your location and receive a list of services that are available in that location which can then help you to retrieve data. But it also means that you do not need to rely on one particular service to discover the data. Instead, the discovery helps to find the services responsible for the data.

Geospatial-centric service discovery ...

Challenge 1 of 3

How to discover services in a location without relying on a discovery service?

This comes with a few challenges on its own. First of all, we want to discover these services without relying on a centralised discovery service. This would completely defeat the purpose of decentralised geospatial data discovery. This also entails that whatever method we do use to discover this data should be as easy as possible for both users and developers.

Geospatial-centric service discovery ...

Challenge 2 of 3

How to interface with the discovered service(s)?

The second challenge is that once we discover services that can provide us with indoor data and positioning systems, we need to collect that data in a way that again does not require specific implementations for each service that is out there, because that would entail that we have to interface with each service in a different way.

Geospatial-centric service discovery ...

Challenge 3 of 3

How to ensure ownership and independence?

The third challenge is related to thw ownership of environements. Since indoor environments are often more private we can assume that the data is possibly also private. Building owners should have the freedom to choose where they publish their information and who can access it.

The first project I briefly wanted to introduce is OpenHPS. OpenHPS is an open source framework for hybrid positioning systems. I will not delve into the details, since it was already covered at a previous FOSDEM talk - but in general you should know that it is a TypeScript framework to develop positioning systems. It is low level, which means that you have a lot of freedom to design your own algorithms, but it also means it is not as easy to use as frameworks that provide you with a bunch of pre-made algorithms and even applications to create an indoor positioning system. The reason why I mention OpenHPS is because it is the foundation of the other projects I will introduce to you today. One issue we noticed after its development was that it was hard to share data between different positioning systems or different frameworks. This is where POSO comes in.
POSO is a linked data ontology to describe positioning systems, the objects they are tracking, the data they produce about those objects, and the environments they are deployed in.
POSO is an ontology that describes positioning systems, the data they produce and the environments they are used in.

SemBeacon

The basic principle of our semantic beacon solution called SemBeacon is a Bluetooth Low Energy beacon that advertises two identifiers and a set of flags. Compatible applications that have no information about this beacon or the data it contains can then ask to retrieve an additional URI that redirects to linked data on the Web containing more information about the beacon, its position and more importantly also its environment. In our semantic data we can make use of existing and extended vocabularies such as the POSO ontology, GeoSPARQL, sensors and observations ontology. By using these existing vocabularies, SemBeacon becomes merely a communication protocol to make nearby devices aware of the data, and what they can expect when they visit the URI.

SemBeacon Bluetooth Specification

BLE 4.X

Our main solution in general consists of an AltBeacon advertisement with a namespace and instance identifier, a set of flags which we will discuss in a moment and an Eddystone-URL compatible scan response. It is the combination of the advertisement data along with the eddystone-url scan response that makes up SemBeacon. Similar to Eddystone-URL we perform encoding of the short URI without further extensions in order to prevent backwards compatibility with existing libraries that can scan for Eddstone.

SemBeacon Bluetooth Specification

Flags

Based on UriBeacon, Bluetooth IPS and Eddystone frames

Bit (MSB) Description Example
0 Indicates if the beacon has a position. 0 = Unsure, 1 = Yes
1 Indicates if the beacon is private. 0 = Public, 1 = Private
2 Indicates if the beacon is attached to a moving object. 0 = No, 1 = Yes
3 Indicates if the beacon has a positioning system. 0 = No, 1 = Yes
4 Indicates if the beacon has observable data. 0 = No, 1 = Yes
5 Indicates if the beacon provides actuable properties. 0 = No, 1 = Yes
6 - 7 Reserved for future use.
Our flags are based on the invisibility flag of UriBeacon and its use cases, the flags and data of the Bluetooth IPS service and finally the use cases of the Eddystone frames. These flags should be configured when setting up a device and do not offer any dynamic information. For example, the position flag would be set to 1 when, during the installation it is clear the beacon will have a fixed position, which would be the case when mounted on wall. Similarly we have other flags that can be decided on during initial setup based on the use case for the SemBeacon. We introduced these flags in order to still provide applications with context on the type of beacon without requiring them to perform an HTTP request. For example, an application that only aims to navigate within a building should not access moveable objects without a position, as these will not be useful.

SemBeacon Bluetooth Specification

BLE 5.X

In addition to our backwards compatible BLE 4 advertisement we also have a work in progress BLE 5 advertisement specification. With the increased payload size for BLE 5 advertisements we can include both the namespace and instance identifiers and uri within a single advertisement. As we no longer have to ensure Eddystone-URL compatibility, we also extended the encoding for the the URI to enable a smaller payload for common linked data URIs, versioning, and a larger URI in general which could negate the need for URL shorteners. BLE 5 advertisements offer better range compared to legacy BLE 4 advertisements which is useful for positioning. Our arduino library already offers the capabilities to use this new service on newer BLE 5 compatible ESP32 hardware for both scanning and advertising. However, our mobile application currently does not scan for BLE 5 advertisements yet.

@openhps/rdf

  • RDF data model for OpenHPS
  • POSO integration
  • Serialisation and deserialisation
const position = new GeographicalPosition();
console.log(RDFSerializer.serialize(position));
Developers are not always eager to want to work with linked data to store information, especially for realtime applications such as positioning systems. This is why we created the @openhps/rdf package. It basically provides a mapping form the OpenHPS data model to RDF and back. This allows developers to easily store their data in a linked data format without specifically needing to know how linked data works. Furthermore, they can query the data is if they are querying a MongoDB database, which is a lot more familiar to most developers. With this project we focus on positioning systems, geospatial data and observational data such as locations.

@openhps/solid

  • Solid storage and access control
  • Solid as a communication broker
With the @openhps/rdf package we already had a way to store our data in a linked data format. However, if we talk about personal data or data that needs to be governed by a user, we also need a proper way to store this data. This is where the Solid project comes into place. For those who are unfamiliar with Solid, it is a project that allows users to store their data in a decentralised way. The data can be binary data, but also structured linked data. Users can assign access rights to the data so that only certain applications or users can access it. In addition to this, Solid offers specifications that enable applications to listen for changes in data - which is perfect for realtime applications such as positioning systems that want to use Solid as a communication broker.
